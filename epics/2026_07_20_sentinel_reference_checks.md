# Plan: MVP Sentinel Reference Checks

Component: `crates/sentinel`, `crates/validator`, new shared crate `crates/safe-tx` (Rust).

---

## Overview

The Safenet Arbitration Charter defines what "secure" means for a Safe transaction: deterministic base guarantees (settings-change blocking, delegatecall integrity — Article IV Part A) and principle-based target-manipulation rules (Article IV Part B). Today, exactly one of Safenet's two transaction-security paths enforces any of this: the validator's direct FROST-signing consensus path runs `crates/validator/src/consensus/checks/mod.rs::check_transaction`, which implements the Part A base guarantees. The Sentinel oracle commit-reveal path (`crates/sentinel`) enforces none of it — `Detector::approve` is a bare destination-address blocklist (`crates/sentinel/src/detector.rs`), with no concept of settings changes, delegatecalls, or target manipulation at all.

This epic builds the **reference Sentinel**: an MVP Sentinel that votes according to the Part A base guarantees and an initial slice of the Part B target-manipulation rules, so it can act as the baseline other (external, more sophisticated) Sentinels are measured against in the Sentinel Game. It has two deliverables: (1) documentation of how "Address Manipulation" (Article I scope: target manipulation) is checked, and (2) a running Sentinel enforcing these guarantees.

**Assumption carried through this whole plan: `epics/2026_07_20_sentinel_vote_reason.md` is already implemented.** That work — adding a `reason` string to a Sentinel's vote, folded into the blind commit hash and surfaced on the onchain `Revealed` event, with `Detector::decide(&self, tx) -> Decision { approve: bool, reason: String }` replacing the old bare-bool `Detector::approve` — is treated as a completed prerequisite, not something this epic builds. Its spec file has already been removed (consistent with that epic being done). **This epic does not touch Solidity, `hashing::commit_hash`, `SentinelRequestState`, `SentinelActionKind::Reveal`, or `SentinelEncoder`** — all of that plumbing for carrying `reason` onchain is assumed to already exist and already work. What this epic adds is the actual check logic that decides `approve` and writes a real, charter-traceable `reason` (rendered from a `RuleId`'s canonical code — see Architecture Decision — without exposing `RuleId` on `Decision` itself) into the field that mechanism already carries.

Building this requires closing two architectural gaps uncovered during investigation, neither of which exists today:
- **No shared crate.** `check_transaction`, its `SafeTransaction`/`Operation` types, and its MultiSend decoder are private to `crates/validator` and typed against validator's own `sol!`-generated structs. `crates/sentinel` independently declares a structurally identical but nominally distinct copy of `SafeTransaction`/`Operation` and has no MultiSend decoder at all. Reusing the base checks in the Sentinel means extracting them first.
- **No onchain data access from `StaticChecker`.** `StaticChecker` is constructed with only a static blocklist; no RPC `Provider`, no persisted per-Safe transaction history. Several Article IV Part B checks (§2.4 Expected target set, address-poisoning comparison) inherently need to compare a transaction's target against the Safe's own prior behavior — that data source does not exist anywhere in the codebase today, for either crate.

Phase 1 closes the shared-crate gap. Phase 2 wires the base guarantees into `Decision` and introduces the charter-mapping. Phases 3–4 ship the excessive-approval check. Phase 5 builds a general remote-check effect — the extension point for whatever isn't implemented locally, including the address-poisoning check's own data needs — and Phase 6 ships that check. Phase 7 applies both against one real, well-known protocol (CoW Swap) as a concrete worked example.

### What "Address Manipulation" means for this MVP

Article IV Part B lists four target-manipulation rules: R-4.3 (value-target), R-4.4 (authorization-target: wrong target), R-4.5 (authorization-target: excessive approval amount), and R-4.6 (known malicious/compromised target). This epic's MVP covers:

- **R-4.5, in full for the deterministic sub-case** (max `uint256` / functionally-unlimited operator approval — §2.5 states this "is always functionally unlimited," no further analysis required). This needs no history or RPC access — pure calldata decoding — so it ships first and cleanly.
- **R-4.3 / R-4.4, narrowed to the address-poisoning pattern** the Charter itself names as circumstantial evidence (§2.4: "the recipient address resembles a prior user address in a way consistent with address poisoning"). Full §2.4 reasoning (protocol-recorded purpose, standard-user-behavior comparisons, dapp metadata) is **explicitly deferred as a general framework** — the Charter's own §2.4 Notes caution that "a new target address is not automatically outside the expected target set," so a naive "deny every non-owner target" implementation would itself violate the Charter. Address-poisoning comparison against the Safe's own recent recipient history is the one sub-case narrow and mechanical enough to ship as a reference check without that broader judgment machinery.
- **A concrete worked example against a real protocol: CoW Swap (Phase 7).** Rather than leave "what would fuller §2.4 reasoning even look like" purely hypothetical, this epic implements it for one well-known, stable integration: recognizing an ERC-20 approval to CoW Protocol's canonical `GPv2VaultRelayer` contract, and checking both that the target is the genuine canonical address (feeding R-4.4's known-good-address set, so first-time interaction with it is never treated as poisoning-suspicious) and that the approved amount is bounded by the sell token's onchain `totalSupply()` rather than merely "not literally unlimited" (a concrete instance of §2.5's "Council considers... token supply" factor). Generalizing this to a broader registry of recognized protocols, or to decoding actual CoW order parameters via offchain order data, remains deferred (see Assumptions) — this phase is one real, fully-worked instance, not a framework.
- **R-4.6 (known malicious/compromised target), in a deliberately crude MVP form.** The existing `SentinelConfig.blocklist` already does exactly this job today, just without any of §2.6/§2.7's source-reliability rigor (no source attribution, no observation timestamp, no reliability basis — just a static address set). Rather than treat it as an unmapped, non-charter special case, this epic **reclassifies the existing blocklist denial as an R-4.6 finding**, explicitly documented as an interim stand-in for real threat-intel integration (a separate, larger follow-up epic — source-reliability policy plus an external HTTP/vendor integration, neither of which exist in either crate today). This keeps the promise that *every* check's denial maps to a charter rule, with the blocklist's evidentiary weakness stated plainly rather than hidden.

## Architecture Decision

**Extract a shared `crates/safe-tx` library crate** owning the canonical `SafeTransaction`/`Operation` types, the MultiSend decoder, and the existing base-guarantee checks (moved from `crates/validator/src/consensus/checks/`, tests included). Both `validator` and `sentinel` depend on it. This turns today's accidental duplication (two independent `sol!` blocks producing the same 12-field struct) into one canonical type, and lets the Sentinel enforce the exact same base guarantees the validator already enforces — not a reimplementation that could drift.

**Introduce a `RuleId` enum as the shared vocabulary between code and Charter**, living in `safe-tx` (so both crates can use it) — but grown incrementally, one variant per check, added in the same PR that implements that check, not all six declared upfront. Phase 2 adds only what it needs (`R4_1SettingsChange`, `R4_2DelegatecallIntegrity`, `R4_6KnownMaliciousTarget`); Phase 4 adds `R4_5ExcessiveApproval` when it ships that check; Phase 6 adds `R4_3ValueTarget`/`R4_4AuthorizationTarget` when it ships poisoning detection. This avoids committing to a rule's exact shape (and having to revisit it) before the check that gives it meaning actually exists. Each variant carries a `code()` method returning its canonical rule code (`"R-4.1"`, etc.) and a doc comment citing the exact charter section. `safe_tx::checks::check_transaction`'s return type changes from `bool` to `Result<(), RuleId>` (`Err(RuleId::R4_1SettingsChange)` / `Err(RuleId::R4_2DelegatecallIntegrity)` as appropriate) — this is the one behavior-preserving-but-signature-changing touch to `check_transaction` in this epic (validator's single call site adapts trivially, `Result::is_ok()` in place of the current bool). Every other check introduced by this epic is expressed the same way, so `StaticChecker::check` can build its `Decision` by collecting the first `Err(rule)` and rendering `reason` from it — the mapping from code to Charter is structural, not a convention someone has to remember to maintain.

**`Decision` itself does not gain a `rule: Option<RuleId>` field.** `RuleId` is an evaluation-time concept, known only while a check runs against a transaction — it is not part of the onchain-facing protocol `Decision` feeds into (commit/reveal, hashing), and keeping it out of `Decision` preserves that boundary. Instead, a denial's `reason` is rendered directly from `RuleId::code()` (just the bare code, e.g. `"R-4.1"` — no longer human-language message for now); `RuleId` itself never leaves the check-evaluation layer. `Decision` stays exactly `{ approve: bool, reason: String }`, unchanged from the vote-reason epic's shape.

**Defer to an operator-configured remote check behind the existing `Effect`/`Resume` mechanism**, not a direct synchronous call. `SentinelService` currently opts out of it entirely (`type Effect = Infallible; type Resume = Infallible;`), and `StateTransition::apply_transition` is meant to stay synchronous, so anything beyond the checks implemented locally in `StaticChecker` needs a round trip through this mechanism. Rather than building a specific, narrow onchain-history fetch as this epic's only remote capability, Phase 5 introduces one general-purpose effect — `Effect::DynamicCheck { safe, transaction }`, resolved by POSTing to an operator-configured HTTPS endpoint and parsing back `{ approve, rule }` — so a sentinel maintainer can plug in whatever check isn't implemented locally (dynamic lists, tracing/simulation, off-chain statistics, address-poisoning history lookups, ...) without forking this crate. The endpoint still cites an existing `RuleId` on denial, so a remote check's decision remains just as charter-traceable as a local one, even though it can legitimately be time-varying/statistical rather than deterministic — that's the whole point of keeping this class of judgment in the Sentinel rather than the validator's FROST-signing path. No endpoint configured means the call resolves to approve immediately, so the reference Sentinel works out of the box with just its local checks. Phase 5 is scoped as its own pair of PRs (5a: the `Effect`/`Resume` types and the HTTPS-calling `RemoteChecker` in isolation; 5b: wiring their emission/consumption into the FSM itself, with its replay semantics addressed explicitly, since `core/src/state/mod.rs` documents that effects may be replayed with a different result — a request whose remote check fails outright is dropped unanswered rather than guessed at either way).

Named accordingly: what was `Detector` (deterministic, local, synchronous — akin to *static* analysis) is renamed to `StaticChecker`, and the `RemoteCheck` effect/resume are renamed to `DynamicCheck`/`DynamicCheckResult` (externally-pluggable, potentially time-varying — akin to *dynamic* analysis); `RemoteChecker` keeps its name as the one concrete implementation of a dynamic check this epic ships.

### Alternatives Considered

- **Duplicate validator's checks into `crates/sentinel` by hand instead of extracting a shared crate.** Rejected: ~450 lines of allow-lists and tests would fork immediately, and any future Article IV Part A change (e.g. a new supported guard/module address) would need updating in two places with no shared test suite to catch drift.
- **Keep `reason` a freeform string with no structured `RuleId`.** Rejected per direct feedback: a reference Sentinel's entire value is that its votes are auditable against the Charter, so the reason must be mechanically traceable to a rule, not just human-readable prose a reader has to interpret. Rendering `reason` from `RuleId::code()` inside each `Check` gets this without exposing `RuleId` on `Decision` itself (see Architecture Decision) or touching the already-implemented onchain-facing plumbing.
- **Treat the existing blocklist as out-of-charter / leave it unmapped.** Rejected: it already makes real approve/deny decisions today, so leaving it un-cited would mean not every check's decision maps to the Charter — the one property this whole redesign is for. Mapping it to R-4.6 (with its evidentiary limitations stated) is honest and keeps the invariant total.
- **Give `StaticChecker` full §2.4 "expected target set" reasoning (protocol-recorded purpose, standard-user-behavior comparisons) in this MVP.** Rejected as too large and too subjective for a first reference implementation — the Charter itself treats this as principle-based/ambiguous, needing precedent to sharpen over time (Article V). Shipping only the address-poisoning sub-case gives a concrete, low-false-positive win now; broader target-set reasoning is a natural follow-up epic once precedent exists.
- **Re-touch the onchain `reveal`/`commit_hash`/`Revealed` plumbing as part of this epic.** Rejected per explicit direction: that work is assumed already done by the (now-removed) `epics/2026_07_20_sentinel_vote_reason.md`. This epic only changes what `StaticChecker` computes, not how a `Decision`'s `reason` gets onchain.

## Tech Specs

### Phase 1 — Extract `crates/safe-tx` (pure refactor)

- New workspace member `crates/safe-tx`, package name `safe-tx`, `edition = "2024"`, depending only on `alloy.workspace = true` (plus `thiserror` if needed for decode errors).
- Move from `crates/validator/src/`:
  - `bindings.rs`'s `SafeTransaction`/`Operation` `sol!` definitions (`bindings.rs:29-53`) → `safe-tx::types`.
  - `consensus/checks/mod.rs` (all of it: `check_transaction`, `check_calls`, `check_self_calls`, `check_delegate_calls`, `check_multi_send`, the local `bindings` sol! block, the `SUPPORTED_*` constants, and the `mod tests` block) → `safe-tx::checks`, unchanged logic and unchanged `bool` return type — the `RuleId` upgrade is Phase 2's job, kept out of this pure-move phase.
  - `consensus/checks/multi_send.rs` → `safe-tx::multi_send`, unchanged.
- `crates/validator`: add `safe-tx.workspace = true` dependency; replace the moved module with a direct call into `safe_tx::checks::check_transaction`; update the one call site (`crates/validator/src/state/transactions.rs:49`) and any imports of `bindings::{SafeTransaction, Operation}` elsewhere in validator to the shared type.
- No behavior change. Existing validator tests (all ~450 lines currently in `checks/mod.rs::tests`) move with the code and must pass unmodified — this is the acceptance bar for the phase, not new test-writing.

### Phase 2 — `RuleId` taxonomy; base guarantees wired into `Decision` (ships "base guarantees" deliverable + the charter-mapping)

- `safe-tx`: add a `RuleId` enum with only the variants this phase needs — `R4_1SettingsChange`, `R4_2DelegatecallIntegrity`, `R4_6KnownMaliciousTarget` — each with a `code()` method returning its canonical citation (`"R-4.1"`, etc.) and a doc comment citing the charter section verbatim. `R4_3ValueTarget`/`R4_4AuthorizationTarget`/`R4_5ExcessiveApproval` are added later, by Phases 4 and 6, alongside the checks that use them — not declared as unused placeholders here. Change `check_transaction`'s return type to `Result<(), RuleId>`; update validator's call site accordingly (mechanical).
- `crates/sentinel`: add `safe-tx.workspace = true`; replace `bindings::consensus::SafeTransaction`/`Operation` (`bindings.rs:64-78`) with the shared `safe_tx::types::SafeTransaction`/`Operation` (open technical question, resolved during implementation: whether alloy's `sol!` can reference a struct from another crate's `sol!` invocation directly, or a small field-by-field conversion is needed — either is fine).
- `Decision` itself is **not** changed to carry a `rule: Option<RuleId>` field — see Architecture Decision for why (`RuleId` is evaluation-time-only, not part of the onchain-facing protocol `Decision` feeds into). Add a small `Check` abstraction (trait or enum) evaluated in a fixed order, first denial wins:
  1. Base guarantees via `safe_tx::checks::check_transaction` → `RuleId::R4_1SettingsChange` / `RuleId::R4_2DelegatecallIntegrity` on failure.
  2. The existing blocklist, reclassified as `RuleId::R4_6KnownMaliciousTarget` (its "static operator list, not source-attributed threat intel" caveat lives in the documentation deliverable below, not in the runtime `reason` string).
- `StaticChecker::check` (already exists) builds its `Decision` by running these checks in order and rendering `reason` as `rule.code()` from the first `Err(rule)`, or `approve: true, reason: String::new()` if all pass. No longer human-language message for now — just the bare code.
- Existing three `StaticChecker` unit tests (`static_checker.rs:26-58`, named `detector.rs` at the time; renamed in Phase 5a's follow-up) updated for the new checks.
- **Documentation deliverable**: the `RuleId` → Charter mapping itself *is* the documentation — each variant's doc comment (`crates/safe-tx/src/rule.rs`) states its charter citation, plain description, and (for R-4.6) the explicit "static blocklist today, real threat-intel later" caveat. No separate reference table: a variant only exists once its check is implemented (per the Architecture Decision's incremental-growth rule), so a hand-maintained "status" column would just restate that invariant and rot the moment it drifts from the enum.

### Phase 3 — Calldata decoding utilities (`safe-tx`, pure decoding, no policy)

- Add to `safe-tx`: decoding for native value transfers (already on `SafeTransaction.value`/`.to`), and calldata decoding for ERC-20 `transfer`/`transferFrom`/`approve`, ERC-721 `approve`/`setApprovalForAll`/`safeTransferFrom`, ERC-1155 `setApprovalForAll`/`safeTransferFrom`/`safeBatchTransferFrom` selectors — extracting the recipient/spender/operator target address and, where applicable, the amount/token-id.
- Must recurse through MultiSend (reusing `safe_tx::multi_send::decode_multi_send`, extracted in Phase 1) so a batched transaction's sub-calls are each decoded individually.
- Output a single enum, e.g. `TargetEffect { recipient: Address, kind: ValueTransfer | Erc20Approval { amount: U256 } | Erc721Operator | Erc1155Operator | ... }`, per sub-call — the shared input Phases 4 and 6's checks consume.
- Pure functions, thoroughly unit-tested against real selector calldata (including at least one MultiSend-batched fixture) — no `StaticChecker`/policy wiring in this phase.

### Phase 4 — R-4.5: excessive/unlimited approval check (ships next slice of "Address Manipulation" + docs)

- `safe-tx`: add the `RuleId::R4_5ExcessiveApproval` variant (not declared before this phase — see Architecture Decision).
- New `Check` for each `TargetEffect::Erc20Approval`/`Erc721Operator`/`Erc1155Operator` decoded via Phase 3: deny with `RuleId::R4_5ExcessiveApproval` if the approval is functionally unlimited per §2.5 — max `uint256` for ERC-20 (deterministic, no further analysis), "approval for all tokens" for ERC-721/1155 operator approvals.
- Phase 2 renders `reason` as the bare `RuleId::code()` (no message); if this phase wants `reason` to state the offending amount/target (e.g. `"unlimited ERC-20 approval to 0x.. (amount = uint256::MAX)"`), it's this phase's job to decide how — `RuleId` still doesn't cross into `Decision`, so any such detail is composed inside the `Check`, not added back onto `Decision` itself.
- Documentation: `RuleId::R4_5ExcessiveApproval`'s own doc comment states the concrete rule (per Phase 2's approach — no separate reference table).

### Phase 5 — Remote-check plumbing (infra only, no built-in policy)

Split into two sub-phases, each its own PR, so the type-level plumbing (mechanically low-risk) and the actual FSM rewiring (the part needing care around replay semantics) land as separately reviewable changes.

Revised from the original "read-only Safe-history effect" framing: rather than building a narrow, Sentinel-owned onchain-history fetch (which would need its own indexing/log-scanning strategy and only ever answers one specific question), Phase 5 introduces a single general-purpose effect that defers to an operator-configured external endpoint. This is deliberately more capable than the epic strictly needs for Phase 6/7's own checks, because it's also the extension point third-party sentinel maintainers are expected to use for whatever isn't (and won't be) implemented in this Rust crate — dynamic blocklists, tracing/simulation calls, statistical scoring, and the like. A remote check's verdict can legitimately change over time (new information, an updated list, a different statistical read) without breaking charter-traceability, because it still cites an existing `RuleId` on denial — the non-determinism lives in *when* a rule fires, not in *whether denials map to the Charter*. That is the intended split of labor between the two transaction-security paths: the validator's FROST-signing path stays 100% deterministic, while the Sentinel is where this kind of statistical/subjective judgment belongs.

**Phase 5a — `Effect`/`Resume` types and the `RemoteChecker`.**
- Add `Effect::DynamicCheck { request_id, safe, transaction }` / `Resume::DynamicCheckResult { request_id, outcome }`, where `outcome` is `Approved | Denied(RuleId) | Failed`.
- Add a `RemoteChecker` (isolated in its own module, sibling to `StaticChecker` rather than folded into it, so the "trigger this endpoint, parse the response" logic stays a single, easily-relocated seam): if no endpoint is configured, resolves to `Approved` immediately, so the reference Sentinel works with just its local checks out of the box; otherwise POSTs `{safe, transaction}` to the configured URL and parses back `{approve, rule}`, validating `rule` against `RuleId::from_code` (added to `safe-tx::rule` for this). Deliberately just a plain HTTPS call issued inline in this crate for now, not a separate service — the interface is the seam to split that out along later, once there's more than one PoC remote check to learn from.
- Wire `SentinelTransition::Effect`/`::Resume` from `Infallible` to these real types, and construct the `RemoteChecker` from a new optional `SentinelConfig.remote_check_url` in `main.rs`/`SentinelService::new` — using the existing `Effect`/`Resume`/`Command::Effect` mechanism (`crates/core/src/state/mod.rs`) rather than a direct synchronous call, a first-of-its-kind use of that mechanism in this codebase.
- No emission or consumption site yet: nothing in `apply_transition` actually produces a `Command::Effect(Effect::DynamicCheck{..})`, so `Message::Resume` is handled as a no-op. Covered by unit tests proving the `RemoteChecker`/`Handler` resolve correctly in isolation from the FSM (including the unconfigured, approved, denied, malformed-response and unreachable-endpoint cases).

**Phase 5b — Wire the effect into the decision flow.**
- In `SentinelTransition::handle_oracle_transaction_proposed`, a local `StaticChecker` denial still decides immediately (no effect needed); otherwise emit `Command::Effect(Effect::DynamicCheck{..})` and move into a new `WaitingForDynamicCheck` state instead of deciding right away.
- On `Message::Resume(Resume::DynamicCheckResult{ request_id, outcome })`: look up the matching `WaitingForDynamicCheck` entry by `request_id` (re-validating against current state before consuming it, the same replay-safety idiom `crates/validator/src/state/sign.rs` already uses, since a resumed effect may legitimately return a different result than the original attempt or arrive after the request has already advanced/been pruned). On `Approved`/`Denied(rule)`, finalize into `WaitingForRequest` as today. On `Failed`, drop the request unanswered rather than guessing at approve/deny — per direct feedback, a Sentinel not responding to a request is an acceptable outcome, whereas voting on an unreachable/malfunctioning remote check would not be.
- No built-in policy in this sub-phase either — covered by integration tests proving the effect resolves correctly end-to-end through a full proposal→decision flow, including the drop-on-failure path.

### Phase 6 — R-4.3/R-4.4: address-poisoning target check (ships remaining "Address Manipulation" scope + docs)

- `safe-tx`: add the `RuleId::R4_3ValueTarget` and `RuleId::R4_4AuthorizationTarget` variants (not declared before this phase — see Architecture Decision).
- New `Check` consuming Phase 3's decoded `TargetEffect`s: for each recipient/spender/operator target, compare it against known-good addresses (Safe's own address, current owners) for poisoning-style similarity (matching prefix/suffix, per §2.4's "resembles a prior user address in a way consistent with address poisoning"). Denies with `RuleId::R4_3ValueTarget` (value transfers) or `RuleId::R4_4AuthorizationTarget` (approvals) as appropriate.
- **Open question carried over from the Phase 5 revision above, to resolve when this phase is actually picked up:** the original design compared a target against the Safe's *prior recipients*, sourced from the (now-removed) Sentinel-owned history fetch. That comparison against the Safe's own transaction history is exactly the kind of dynamic, potentially-statistical lookup Phase 5's `DynamicCheck` effect exists for — so this phase may ship as a local `Check` restricted to the addresses already known synchronously (Safe address, current owners) with the fuller history-aware comparison left as a reference `RemoteChecker` responder implementation, rather than reintroducing a Sentinel-owned history fetch. Decide this once Phase 5 has landed and there's a concrete `RemoteChecker` responder to learn from.
- Per the Charter's own §2.4 Notes, a target merely being *novel* is not itself a denial — only a *poisoning-pattern match against a known-good address* denies. A novel-but-dissimilar address is approved by this check (deferring the fuller §2.4 standard-user-behavior reasoning to a later epic, as noted in Overview).
- Documentation: `RuleId::R4_3ValueTarget`/`R4_4AuthorizationTarget`'s doc comments state the poisoning heuristic — what similarity means precisely, what data it's compared against, and its known limitation (does not yet implement full §2.4 reasoning; new-but-legitimate counterparties are approved, consistent with the Charter).

### Phase 7 — Worked example: CoW Swap approval sanity (concrete R-4.4/R-4.5 instance)

- Add a small, static "recognized protocol" registry to `safe-tx` (same style as the existing `SUPPORTED_FALLBACK_HANDLERS`/`SUPPORTED_GUARDS` constant tables): CoW Protocol's `GPv2VaultRelayer` contract address per supported network (Ethereum, Arbitrum, Gnosis Chain — the Charter's own network scope, Article I). Deliberately narrow and hardcoded — one real example, not a generalized allowlist framework.
- Feed this registry into Phase 6's known-good-address set: a decoded `TargetEffect::Erc20Approval`/operator-approval targeting a recognized address is never flagged by the poisoning check (`RuleId::R4_4AuthorizationTarget`) as "novel," regardless of the Safe's own transaction history — `GPv2VaultRelayer` is a canonical, network-wide contract any Safe may legitimately interact with for the first time, and first-time interaction with it is not itself suspicious.
- Tighten the R-4.5 (`RuleId::R4_5ExcessiveApproval`) bound specifically for approvals to this recognized target: in addition to Phase 4's flat "max `uint256` is always unlimited" rule, deny if the approved amount exceeds the sell token's `totalSupply()` — a concrete, worked instance of §2.5's "Council considers... token supply" factor, rather than leaving that factor abstract. Same open question as Phase 6: whether `totalSupply()` is looked up as a reference `RemoteChecker` responder (reusing Phase 5a's effect, no chain `Provider` inside the Sentinel itself) or via a narrow, dedicated onchain read — decide alongside Phase 6's.
- Depends on Phases 4, 5b, and 6 (reuses the `RuleId`s, decoding, and known-good-address set they introduce — no new `RuleId` variants).
- Explicitly out of scope for this phase: recognizing any protocol beyond CoW Swap, and decoding an actual CoW order's parameters (e.g. via its offchain API to compare against a specific order's `sellAmount`) — both are left to the follow-up epic that takes on general §2.4 reasoning (Assumptions).
- Documentation: extend `RuleId::R4_4AuthorizationTarget`'s and `R4_5ExcessiveApproval`'s doc comments with this worked CoW Swap example.

### Phase 8 — Remove this plan

Delete `epics/2026_07_20_sentinel_reference_checks.md` once Phases 1–7 are merged.

## Assumptions

- **`epics/2026_07_20_sentinel_vote_reason.md` is treated as already implemented and is out of scope here.** `Detector::check -> Decision { approve, reason }` and the onchain threading of `reason` through commit/reveal already exist; this epic does not modify Solidity, `hashing::commit_hash`, `SentinelRequestState`, `SentinelActionKind::Reveal`, or `SentinelEncoder`, nor does it change `Decision`'s shape. It only changes what checks populate `reason` with (rendered from an evaluation-time-only `RuleId`, which is never itself added to `Decision` — see Architecture Decision).
- **R-4.6 is covered only in its crudest form (the existing static blocklist, reclassified).** Real source-reliability evaluation (§2.6/§2.7) and threat-intel integration are left for a future epic — this MVP only guarantees that the blocklist's denials are honestly labeled as R-4.6, not that R-4.6 is fully implemented.
- **No new database schema, and no chain `Provider` inside `SentinelService`.** Phase 5's `DynamicCheck` effect is a plain HTTPS call to an operator-configured endpoint, not a persisted local index or a direct RPC read — whatever onchain (or offchain) data a given dynamic check needs is that endpoint's own concern, not this crate's.
- **General §2.4 "expected target set" reasoning (protocol-recorded purpose, standard-user-behavior comparison, dapp metadata, as a scalable framework covering arbitrary protocols) is explicitly deferred; one concrete instance of it (CoW Swap) is not.** This MVP's R-4.3/R-4.4 coverage is the address-poisoning sub-case (Phase 6) plus, as a real worked example, Phase 7's CoW Swap recognized-target/token-supply-bounded-amount check. What's still deferred to a follow-up epic: recognizing *other* protocols (Uniswap, Aave, etc. — each needs its own recognizer and its own notion of "sane limits and addresses," which doesn't scale the way the protocol-agnostic poisoning heuristic does), and decoding an actual CoW order's parameters (e.g. via CoW's offchain API to compare against that specific order's `sellAmount`, rather than Phase 7's coarser token-`totalSupply()` bound) — both would need machinery (an offchain HTTP client, a general recognized-protocol registry) this epic doesn't build.
- **Concrete check algorithms (exact poisoning-similarity heuristic, exact decoded-selector list, exact `Check`/`RuleId` Rust shapes) are intentionally left loose in this spec** per the epic's own notes — to be nailed down during each phase's implementation/PR review, not gated on this planning doc.
